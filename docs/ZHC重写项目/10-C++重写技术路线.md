## 10.3 MVP 优先：可视化追踪先行

> 基于 AI 接入报告.xml 的建议：**MVP 只做一件事，用现有的 IR 优化层，新增一个追踪模式。**

```
zhc run --trace 输入文件.zhc
输出: trace.json + trace.html
预计开发量：约 580 行代码，1-2 周完成。
```

**为什么可视化追踪优先于 AI 接入？**

| 因素 | 可视化追踪 | AI 编程接口 |
|:---|:---|:---|
| **开发量** | ~580 行代码 | ~200h |
| **验证速度** | 1-2 周 | 2 个月 |
| **用户反馈** | 老师/学生立即可用 | 需要 API 配置 |
| **差异化** | 中文可视化执行，独一无二 | 竞品较多 |
| **技术依赖** | 仅依赖现有 IR | 依赖 LLM API |

## 10.5 各阶段 Go/No-Go 检查点

> **强制规则**: 阶段验收未通过 → 不得自动进入下一阶段 → 必须先解决当前问题

| 检查点 | 通过标准（定量） | 失败处理 |
|:---|:---|:---|
| **Phase 0 → 1** | `zhc run --trace fibonacci.zhc` 生成 trace.html；JSON 格式验证通过 | Phase 0 最多延长至 3 周；仍未通过则降级为"仅生成 JSON"模式 |
| **Phase 1 → 2** | C++ Lexer + Parser 跑通现有 Python 版本测试集 ≥ 90% | 停 1 周集中修 Bug；Parser 错误率 < 5% 才放行 |
| **Phase 2 → 3** | 能编译运行 fibonacci.zhc（递归），输出正确 | 停 2 周修 CodeGen Bug；必要时加人工 Code Review |
| **Phase 3 → 4** | trace.html 含变量追踪 + 调用栈，浏览器正常渲染 | 修复 HTML 生成器；问题严重则降级为 JSON-only 模式 |
| **Phase 4 → 5** | AI 错误解释成功率 > 80%，P95 响应时间 < 5s | 增加超时处理；增加本地模型作为降级选项 |
| **Phase 5 → 6** | 沙箱执行不崩溃（成功率 > 95%） | 增加沙箱稳定性测试；问题严重则推迟 Phase 5 |
| **Phase 6 → v1.0** | .zhc/.c 各 3 个测试用例通过；性能基准 ≤ Clang 2x | Python 前端可推迟；按优先级先保证 .zhc + .c |
