# 第十六章：当前技术债务清单

**版本**: v2.0 | **日期**: 2026-04-12

---

## 16.1 技术债务概览

> **技术债务** = 走过的捷径 + 遗留的未解问题 + 为未来挖的坑  
> **清理成本** = 修复工时 + 回归测试工时 + 文档更新工时

### 债务严重程度分级

| 级别 | 标识 | 含义 | 修复策略 |
|:---|:---:|:---|:---|
| **A 级** | 🔴 | 严重阻塞 C++ 重写，或有数据损坏风险 | C++ 重写时必须修复 |
| **B 级** | 🟠 | 功能缺陷或长期维护负担 | C++ 重写时一并解决 |
| **C 级** | 🟡 | 体验问题或长期技术负担 | 视时间决定 |
| **D 级** | 🟢 | 低优先级债务 | 有则改之，无则加勉 |

## 16.2 前端技术债务（A + B 级）

### 🔴 A-01: Lexer 状态机不完整

| 项目 | 内容 |
|:---|:---|
| **债务描述** | 字符串字面量只支持单字节编码，中文字符串会截断或乱码 |
| **影响范围** | 中文编程体验，中文注释，中文字符串常量 |
| **债务根源** | 早期实现时未考虑 Unicode，只测了 ASCII |
| **当前规避** | 注释内可写中文（Python 层处理），但字符串常量不行 |
| **修复工时** | 24h（需要重新设计 SourceManager 编码层） |
| **C++ 修复方案** | SourceManager 统一用 UTF-8 + llvm::MemoryBuffer，Lexer 返回 UTF-8 字节序列 |

---

### 🔴 A-02: Parser 错误恢复能力几乎为零

| 项目 | 内容 |
|:---|:---|
| **债务描述** | 一个语法错误会导致整页诊断信息刷屏，用户无法定位真正问题 |
| **影响范围** | 所有用户的错误体验 |
| **债务根源** | Parser 用的是"遇到错误直接 panic"的策略，没有同步点 |
| **当前规避** | 靠用户提供最小复现用例 |
| **修复工时** | 40h（需要实现 PANIC/SYNC 错误恢复机制） |
| **C++ 修复方案** | 参考 Clang 的"同步点（synchronization points）"策略，错误后跳到分号/右大括号继续 |

---

### 🟠 B-01: AST 节点继承层次混乱

| 项目 | 内容 |
|:---|:---|
| **债务描述** | `ast_nodes.py` 中 76+ 节点类型，部分继承链不清晰（如 FunctionDecl 和普通 Decl 混在一起） |
| **影响范围** | 新增语法特性时需要修改多个类，维护成本高 |
| **债务根源** | 早期快速迭代时没有设计好继承体系 |
| **修复工时** | 32h（重新梳理继承关系） |
| **C++ 修复方案** | C++ 强制类型系统 + 虚函数，直接显式建模继承层次 |

---

### 🟠 B-02: Token 和 AST 节点位置信息丢失

| 项目 | 内容 |
|:---|:---|
| **债务描述** | Lexer 返回 Token 时丢失源码行/列信息，导致错误定位不准确 |
| **影响范围** | 所有错误消息行号不准 |
| **债务根源** | Token 定义时没有加 source_location 字段 |
| **修复工时** | 16h（Token 加位置字段，全链路透传） |
| **C++ 修复方案** | `llvm::SMLoc` + `llvm::SourceMgr` 统一管理，所有 AST 节点持有 SourceRange |

---

## 16.3 语义分析技术债务（A + B 级）

### 🔴 A-03: 类型推导在闭包场景下崩溃

| 项目 | 内容 |
|:---|:---|
| **债务描述** | 在闭包中使用外部变量时，类型推导进入死循环或返回 `None` |
| **影响范围** | 所有使用闭包的项目 |
| **债务根源** | 闭包的捕获列表推导有循环依赖：外部变量类型依赖闭包，闭包类型依赖外部变量 |
| **当前规避** | 手动给闭包捕获的变量加类型注解 |
| **修复工时** | 48h（需要实现两轮推导：第一轮收集，第二轮求解） |
| **C++ 修复方案** | 分两阶段语义分析：声明阶段 + 主体阶段，闭包捕获分析独立遍历 |

---

### 🔴 A-04: 泛型实例化有内存泄漏

| 项目 | 内容 |
|:---|:---|
| **债务描述** | 模板实例化后的 IR 节点没有正确释放，编译大型项目时内存持续增长 |
| **影响范围** | 编译超过 100 个泛型实例时内存超过 1GB |
| **债务根源** | Python 没有 RAII，IR 节点手动管理但析构函数调用不完整 |
| **当前规避** | 大型项目分段编译 |
| **修复工时** | 24h（加引用计数或改用弱引用） |
| **C++ 修复方案** | C++ unique_ptr/shared_ptr 自动管理内存 |

---

### 🟠 B-03: 符号表不支持向前引用

| 项目 | 内容 |
|:---|:---|
| **债务描述** | 结构体/函数在定义之前被使用时报错，无法写出"先使用后定义"的代码风格 |
| **影响范围** | 习惯先声明接口后写实现的用户 |
| **债务根源** | 符号表用单遍解析，第一遍看到未定义符号就报错 |
| **修复工时** | 20h（符号表加 placeholder，后续填坑） |
| **C++ 修复方案** | 两遍语义分析，第一遍只填符号表，第二遍检查类型 |

---

### 🟠 B-04: const 传播分析不完整

| 项目 | 内容 |
|:---|:---|
| **债务描述** | 条件分支中的 const 变量出了分支后仍然被当作 const，导致优化失败 |
| **影响范围** | 需要深度优化的项目 |
| **债务根源** | 常量传播算法没有处理分支合并的 phi 节点 |
| **修复工时** | 16h |
| **C++ 修复方案** | LLVM 已有 GVN Pass 的常量传播，直接复用 |

---

## 16.4 IR 生成技术债务（A + B 级）

### 🔴 A-05: IR 不是严格 SSA 形式

| 项目 | 内容 |
|:---|:---|
| **债务描述** | ZhC IR 中的寄存器会被多次赋值，限制了优化 Pass 的效果 |
| **影响范围** | 所有需要 SSA-based 优化的场景 |
| **债务根源** | 生成 IR 时直接复用寄存器编号，没有做 SSA 转换 |
| **当前规避** | 依赖后端 gcc/clang 做 SSA 化 |
| **修复工时** | 80h（实现完整 SSA 转换） |
| **C++ 修复方案** | 直接生成 LLVM IR，LLVM 原生 SSA，不需要自己实现 |

---

### 🔴 A-06: 异常处理的 IR 生成不完整

| 项目 | 内容 |
|:---|:---|
| **债务描述** | `尝试/捕获` 语法生成的 IR 无法正确展开，运行时崩溃 |
| **影响范围** | 使用异常处理的任何程序 |
| **债务根源** | 异常处理 IR 需要维护 landingpad + personality 函数，代码生成时跳过了 |
| **当前规避** | 用户不使用异常处理语法 |
| **修复工时** | 56h（需要研究 Clang 的异常处理 IR 生成） |
| **C++ 修复方案** | 参考 Clang 的 exception handling IR，复用 LLVM 的 EH 基础设施 |

---

### 🟠 B-05: 函数调用的 ABI 参数传递不统一

| 项目 | 内容 |
|:---|:---|
| **债务描述** | 调用约定（cdecl/stdcall/fastcall）硬编码，跨平台时参数传递顺序错误 |
| **影响范围** | Windows x64 平台编译失败 |
| **债务根源** | 只测试了 Linux x64/ARM64，没有处理 Windows x64 的 shadow space |
| **修复工时** | 24h |
| **C++ 修复方案** | 通过 LLVM TargetMachine 获取平台 ABI，自动生成正确的 calling convention |

---

## 16.5 运行时技术债务（B + C 级）

### 🟠 B-06: 协程栈空间固定，无法动态增长

| 项目 | 内容 |
|:---|:---|
| **债务描述** | 协程的栈大小固定 64KB，递归调用超过限制时直接崩溃 |
| **影响范围** | 使用协程且调用链较深的用户 |
| **债务根源** | 协程实现用了固定大小的栈池 |
| **修复工时** | 40h（需要实现栈切换或分段栈） |
| **C++ 修复方案** | 参考 LLVM 的 split stack 或完全用 async 函数 |

---

### 🟡 C-01: 反射系统的实现是存根

| 项目 | 内容 |
|:---|:---|
| **债务描述** | `类型信息` 和 `成员信息` 关键字存在但返回空数据 |
| **影响范围** | 需要元编程的用户 |
| **债务根源** | 快速发布时砍掉的特性，留了接口 |
| **修复工时** | 32h |
| **C++ 修复方案** | 在编译时生成 RTTI 结构体，运行时通过符号查找 |

## 16.8 C++ 重写债务清理策略

```
Python 版已知 Bug（不修）
         │
         ▼
    ┌────────────┐
    │  A 级债务  │──────→ C++ 重写内置解决（272h）
    └─────┬──────┘
          │
    ┌─────▼──────┐
    │  B 级债务  │──────→ C++ 重写时一并解决（已含在阶段工时）
    └─────┬──────┘
          │
    ┌─────▼──────┐
    │  C 级债务  │──────→ v1.0 之后视情况处理
    └────────────┘
```

**关键结论**: C++ 重写不仅是性能升级，也是**债务清理**。272h 的 A 级债务在 Python 版本里修了还会再犯，只有 C++ 的类型系统和 RAII 能从根本上解决。
