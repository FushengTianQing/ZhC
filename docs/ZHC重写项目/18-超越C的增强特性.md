# 第十八章：超越 C 的增强特性（参考 Rust 改进路线）

**版本**: v2.0 | **日期**: 2026-04-13

---

## 18.0 设计原则

> **ZhC 的定位不是 C 的完全替代品，而是 C 的安全增强方言。**  
> 目标：在保留 C 的性能和底层控制能力的同时，引入编译期安全检查，让常见错误（空指针、内存泄漏、数据竞争）在编译期被发现而不是运行时崩溃。

### 三条铁律

| 铁律 | 含义 | C 的问题 | ZhC 的解法 |
|:---|:---|:---|:---|
| **Safe by Default** | 安全是默认行为，不是可选项 | `int* p = NULL; *p = 0;` 编译通过，运行崩溃 | 可空指针默认追踪，空解引用编译报警 |
| **Explicit over Implicit** | 隐式转换和默认初始化全部禁止 | `int x;` 随机值，调试半天才发现 | `整数型 x;` 强制要求初始化或显式赋初值 |
| **No Cost Abstraction** | 增强特性零运行时开销 | `#define PI 3.14` 无类型安全 | `常量 PI: 浮点型 = 3.14` 类型安全，编译期求值 |

### 增强特性选型原则

Rust 对 C 的改进有几十项，ZhC 不会照搬全部，而是选择**编译期可检查、不引入 GC、不破坏 C ABI、且对中文学习者有实际教育价值**的特性，按优先级分为三级：

| 优先级 | 特性类型 | 选择标准 |
|:---:|:---|:---|
| **E0** | 编译期必做 | C 的**未定义行为**（UB），学习者高频踩坑 |
| **E1** | 语言增强 | Rust 的招牌特性，对 C 有实质改进 |
| **E2** | 高级特性 | 有则锦上添花，无则不影响 v1.0 发布 |

## 18.2 E0 级：编译期必做的安全增强

> **核心原则：把 C 的 7 种未定义行为（UB）变成编译期错误。**

### 18.2.1 空指针安全（对应 Rust 的 Option<T>）

**C 的问题**：
```c
int* p = NULL;
*p = 42;  // 编译通过，运行崩溃，未定义行为
```

**Rust 的解法**：
```rust
let p: Option<&i32> = None;
*p.unwrap();  // 编译错误：called `Option::unwrap()` on a `None` value
```

**ZhC 的实现**：引入 `?空型` 标记可空指针

```zhc
// ZHC：普通指针默认非空，编译期保证
整数型* ptr = 取地址(x);  // 编译通过，非空指针

// ZHC：可空指针必须显式用 ?空型 标记
?空型 整数型* nullable = 空指针;  // 可空指针必须显式声明

// ZHC：对可空指针解引用，编译器强制空检查
如果 nullable.是空() {
    // 空指针处理分支（必须处理）
}
否则 {
    整数型 val = nullable.取数据();  // 只有非空才能取数据
}

// ZHC：直接解引用可空指针 → 编译错误
*nullable = 42;  // ❌ 编译错误：可空指针不能直接解引用
```

**实现方案**：
- 类型系统中新增 `?空型<T>` 包装类型（对应 Rust 的 `Option<T>`）
- 编译器对 `?空型` 类型的解引用生成**空指针检查 IR**（`br isnull label_ crash`）
- 配合 AI 监控层：如果运行时真的触发空指针，AI 分析并给出修复建议

**工时**：80h（类型系统 + Parser + Sema + IR 生成）

### 18.2.2 强制初始化检查（对应 Rust 的默认值 + Drop 绑定）

**C 的问题**：
```c
int x;  // 未初始化，随机值
printf("%d", x);  // UB
```

**Rust 的解法**：
```rust
let x: i32;  // 编译错误：cannot find value `x`
let x: i32 = 0;  // 正确：必须初始化
```

**ZhC 的实现**：默认要求初始化，未初始化变量禁止使用

```zhc
// ZHC：未初始化变量 → 编译警告（可配置为错误）
整数型 x;  // ⚠️ 警告：变量 x 未初始化，建议初始化

// ZHC：未初始化变量直接使用 → 编译错误
打印(x);  // ❌ 编译错误：使用了未初始化变量 x

// ZHC：正确用法
整数型 x = 0;
整数型 y = 获取值();
```

**实现方案**：
- Sema 分析阶段对每个局部变量建立「已初始化」标记
- 控制流分析：如果变量在所有路径上都被初始化过，则标记为安全
- 提供编译选项：`-Werror=uninitialized`（将警告升级为错误）

**工时**：40h（Sema 初始化追踪 + 控制流分析）

### 18.2.3 数组边界检查（对应 Rust 的数组索引）

**C 的问题**：
```c
int arr[10];
arr[20] = 5;  // 越界写入，破坏其他内存，UB
```

**Rust 的解法**：
```rust
let arr = [1, 2, 3];
arr[20];  // 编译错误：index out of bounds
```

**ZhC 的实现**：分两种模式，兼顾性能

```zhc
// ZHC：静态数组 → 编译期边界检查
整数型 arr[10] = [1, 2, 3, 0, 0, 0, 0, 0, 0, 0];
arr[20] = 5;  // ❌ 编译错误：数组长度 10，索引 20 越界

// ZHC：动态数组（运行时分配）→ 运行时边界断言
空型* dyn_arr = 申请内存(10 * sizeof(整数型));
dyn_arr[20] = 5;  // ⚠️ 编译警告 + 运行时断言 abort()

// ZHC：显式跳过检查（仅在学习阶段使用）
整数型* raw = 强制转换(dyn_arr, 整数型*);
raw[20] = 5;  // ✅ 显式跳过检查，用户自己负责
```

**实现方案**：
- 静态数组：编译期常量索引直接计算，超界报错
- 动态数组：IR 生成时插入 `llvm.assert` 或 `__zhc_bounds_check` 函数
- 提供 `-fno-bounds-check` 编译选项用于性能敏感代码

**工时**：48h（Parser + Sema + IR 生成 + 测试）

### 18.2.4 整数溢出检测（对应 Rust 的溢出模式）

**C 的问题**：
```c
int x = INT_MAX;
x = x + 1;  // UB（补码溢出行为由编译器决定）
```

**Rust 的解法**：
```rust
// Debug 模式：panic
// Release 模式：wrapping
let x: i32 = i32::MAX;
x.checked_add(1);  // 返回 None 而不是 UB
```

**ZhC 的实现**：溢出模式可配置，默认检测

```zhc
// ZHC：默认模式（检测溢出）
整数型 x = 2147483647;
x = x + 1;  // ⚠️ 编译警告：整数型溢出，建议使用 溢出检查 加法
x = 溢出检查(x + 1);  // ✅ 显式溢出检查，溢出时程序终止

// ZHC：编译选项可切换溢出行为
// zhc compile --overflow=wrap   // 补码环绕
// zhc compile --overflow=saturate // 饱和到最大/最小值
// zhc compile --overflow=panic  // 溢出时 panic
```

**实现方案**：
- IR 生成阶段对加/减/乘运算注入溢出检查 Intrinsic（`llvm.sadd.with.overflow.i32`）
- LLVM 优化 Pass 可以根据编译选项决定是保留检查还是转为 wrapping
- 编译器警告：`--warn-overflow` 将溢出警告默认打开

**工时**：32h（IR 生成 + 编译选项 + 测试）

### 18.2.5 使用已释放内存检测（对应 Rust 的借用检查）

**C 的问题**：
```c
int* p = malloc(sizeof(int));
free(p);
*p = 42;  // Use-after-free，编译通过，运行时 UB
```

**Rust 的解法**：借用检查器（Borrow Checker）静态保证

**ZhC 的实现**：作用域 + RAII 自动析构

```zhc
// ZHC：指针离开作用域后自动释放，析构函数自动调用
空型* func() {
    整数型* p = 申请内存(sizeof(整数型));  // 申请内存

    如果 (p == 空指针) {
        返回 空指针;  // p 在此返回前已释放（RAII）
    }

    *p = 42;

    返回 p;  // ⚠️ 警告：p 指向的内存被返回，但指针本身即将离开作用域
             // 编译器提示：所有权转移或复制指针
}
// ⚠️ 警告：函数返回后，p 的所有权未明确转移
// 建议：使用 共享指针 或 独享指针 返回

// ZHC：正确用法：显式所有权转移
独享指针(整数型) p = 创建独享指针(整数型, 42);
返回 p.取所有权();  // 所有权转移，p 不再有效
```

**实现方案**：
- 利用已有的 RAII 基础设施（ZhC v6.0 已有 UniquePtr/SharedPtr）
- Sema 增加**生命周期分析**：对返回指针的函数检查所有权是否明确转移
- AI 监控层在运行时检测 `free()` 后的使用（第二道防线）

**工时**：40h（Sema 生命周期分析 + 所有权检查）

### 18.2.6 野指针检测（对应 Rust 的生命周期）

**C 的问题**：
```c
int* dangling() {
    int x = 10;
    return &x;  // 返回局部变量地址，悬垂指针（Dangling Pointer）
}
```

**Rust 的解法**：
```rust
fn dangling() -> &i32 {  // 编译错误：
    let x = 10;          // `x` doesn't live long enough
    &x                    // borrowed value does not live long enough
}
```

**ZhC 的实现**：返回指针必须标注生命周期

```zhc
// ZHC：返回局部变量地址 → 编译错误
整数型* dangling() {
    整数型 x = 10;
    返回 取地址(x);  // ❌ 编译错误：x 的生命周期在函数返回时结束，
                    //     返回的指针是悬垂指针（Dangling Pointer）
}

// ZHC：接受外部生命周期参数
整数型* copy_value(整数型* 输入, 生命周期 'a) -> 生命周期 'a 整数型* {
    静态 整数型 buffer;
    buffer = *输入;
    返回 取地址(buffer);  // ❌ 仍然错误：buffer 是静态的，但返回后不安全
}

// ZHC：正确：返回堆分配内存或参数内存
整数型* safe_ptr(整数型* 输入) {
    返回 输入;  // ✅ OK：返回参数内存，生命周期的输入端
}

// ZHC：生命周期标注（参考 Rust lifetime）
// 语法：生命周期 'a 表示"至少 'a 这么长"
// &'a 整数型 表示"生命周期至少为 'a 的整数型引用"
```

**实现方案**：
- 引入**生命周期标注**（lifetime annotation）语法：`生命周期 'a`
- Sema 中实现生命周期分析（参考 Rust NLL 设计）
- 初期可以简化版：禁止返回局部变量地址，要求显式堆分配

**工时**：64h（生命周期语法解析 + Sema 分析 + IR 集成）

### 18.2.7 NULL 返回强制处理（对应 Rust 的 ? 操作符）

**C 的问题**：
```c
FILE* f = fopen("test.txt", "r");
// 直接使用 f，忘记检查 NULL，fprintf(f, ...) 可能崩溃
```

**Rust 的解法**：
```rust
let f = File::open("test.txt")?;  // ? 操作符：如果是 Err 则提前返回
```

**ZhC 的实现**：NULL 返回函数必须显式处理

```zhc
// ZHC：可能返回空的函数，调用时必须处理空情况
文件型* f = 打开文件("test.txt", "r");
// ❌ 编译错误：打开文件 返回 ?空型，调用必须处理空情况

// ZHC：必须显式处理空返回
文件型* f = 打开文件("test.txt", "r") ?;
如果 (f.是空()) {
    打印("文件打开失败");
    退出(1);
}
打印文件内容(f);

// ZHC：使用 ? 操作符简化空检查（类似 Rust 的 ?）
// 注意：ZhC 中 ? 用于可空指针的安全解引用
文件型* f = 打开文件("test.txt", "r") ?
    .读取全部()
    .打印();
```

**实现方案**：
- `?` 语法糖：编译期转换为 `if (ptr == NULL) return NULL/return;`
- Sema 检查：调用返回 `?空型<T>` 的函数时，必须有空处理路径
- AI 辅助：AI 自动插入正确的空检查代码

**工时**：32h（`?` 语法糖 + Sema 空处理检查）

## 18.3 E1 级：语言增强

### 18.3.1 穷举式 switch（对应 Rust 的 match 穷举检查）

**C 的问题**：
```c
enum Color { RED, GREEN, BLUE };
switch (c) {
    case RED: return 1;
    case GREEN: return 2;
    // 漏写了 BLUE，编译通过，但逻辑错误
}
```

**Rust 的解法**：
```rust
match color {
    RED => 1,
    GREEN => 2,
    // 编译错误：non-exhaustive patterns
}
```

**ZhC 的实现**：
```zhc
枚举 颜色 { 红, 绿, 蓝 }

颜色 c = 颜色.红;

// ZHC：switch 默认穷举检查（有默认分支）
选择 (c) {
    当 颜色.红: 返回 1;
    当 颜色.绿: 返回 2;
    默认: 返回 0;  // ✅ 显式穷举，即使有默认分支也会提示其他分支未处理
}

// ZHC：严格模式（无 默认: 分支时强制穷举所有枚举值）
选择 (c) {
    当 颜色.红: 返回 1;
    当 颜色.绿: 返回 2;
    // ❌ 编译错误：枚举 颜色 未穷举
    //    缺少分支：当 颜色.蓝
}

// ZHC：穷举所有分支后正确
选择 (c) {
    当 颜色.红: 返回 1;
    当 颜色.绿: 返回 2;
    当 颜色.蓝: 返回 3;
    // ✅ 穷举后正确：不需要写 默认: 分支，编译器确保所有情况覆盖
}
```

**工时**：24h（Sema 穷举检查）

### 18.3.2 结果型 Result（对应 Rust 的 Result<T, E>）

**Rust 的解法**：
```rust
fn read_file() -> Result<String, io::Error> {
    let mut f = File::open("foo.txt")?;
    let mut s = String::new();
    f.read_to_string(&mut s)?;
    Ok(s)
}
```

**ZhC 的实现**：
```zhc
// ZHC：结果型，强制错误处理
结果型(整数型, 错误码) 安全的除法(整数型 a, 整数型 b) {
    如果 (b == 0) {
        返回 结果型.错误(错误码.除零错误);
    }
    返回 结果型.正常(a / b);
}

// ZHC：调用方必须处理错误
结果型(整数型, 错误码) r = 安全的除法(10, 0) ?;
匹配 r {
    情况 正常(值): 打印("结果是 ", 值);
    情况 错误(原因): 打印("错误: ", 原因);
}

// ZHC：try 操作符（错误自动传播）
// 注意：ZhC 的 try 语义与 C++ 不同，用于 Result 的错误传播
```

**工时**：40h（Result 类型系统 + 匹配表达式 + 错误传播）

### 18.3.3 线程安全标注（对应 Rust 的 Send/Sync trait）

**C 的问题**：
```c
// 两个线程同时访问同一块内存，无任何保护 → 数据竞争（UB）
```

**Rust 的解法**：
```rust
// 编译器静态检查：非 Send 类型跨线程传递会编译错误
let data = std::rc::Rc::new(1);
std::thread::spawn(move || { ... });  // 编译错误：Rc 不是 Send
```

**ZhC 的实现**：
```zhc
// ZHC：线程安全类型标注
共享型 整数型 shared_count = 0;  // 共享型 = 线程间共享（带锁）
线程独享型 整数型 local_count = 0;  // 线程独享型 = 不可跨线程传递

// ZHC：互斥锁类型（编译期检查）
互斥锁(整数型) counter = 创建互斥锁(整数型, 0);

// ZHC：Send/Sync trait 标注（E1 后期实现）
// 实现者标记 类型.可发送 = 真 允许跨线程传递
```

**工时**：32h（线程安全类型 + 互斥锁 IR 生成）

### 18.3.4 格式化字符串安全（对应 Rust 的 format! 宏）

**C 的问题**：
```c
printf("%s", 42);  // UB：参数类型不匹配
printf(user_input);  // 格式化字符串漏洞（攻击向量）
```

**Rust 的解法**：
```rust
println!("{}", 42);  // 编译期检查格式参数类型
println!("{}", "hello");  // OK
println!(user_input);  // 编译错误：unsafe
```

**ZhC 的实现**：
```zhc
// ZHC：中文格式化函数，编译期检查参数类型
打印("数字:", 42);  // ✅ OK，参数是整数型，格式化串无占位符则忽略参数
打印("{} + {} = {}", 1, 2, 3);  // ✅ OK，参数类型自动匹配
打印("{}", 42);  // ✅ OK，通用占位符

// ZHC：格式化漏洞防护
打印("用户输入: {}", user_input);  // ✅ OK，user_input 作为数据而非格式化串
打印(用户输入);  // ❌ 编译警告：可疑的纯字符串打印，可能有格式化漏洞

// ZHC：中文格式化占位符
打印("我叫{}，今年{}岁", 名字, 年龄);
打印("计算结果：{:d}", 数字);  // :d 显式整数型
打印("浮点数：{:.2f}", 圆周率);  // :.2f 两位小数
```

**工时**：24h（格式化检查 Pass + 测试）

## 18.6 与 Rust 的对比定位

| 特性 | Rust | ZhC | 差异 |
|:---|:---|:---|:---|
| **内存安全模型** | 所有权 + 借用（编译期，无 GC） | 生命周期追踪 + RAII（编译期，无 GC） | ZhC 更轻量，Rust 更严格 |
| **空安全** | `Option<T>` + `None` | `?空型<T>` + `空指针` | 语法不同，理念一致 |
| **错误处理** | `Result<T, E>` + `?` | `结果型<T, E>` + `?` | 几乎相同 |
| **模式匹配** | `match` 穷举 | `选择`/`匹配` 穷举 | 语法不同，理念一致 |
| **数组安全** | 编译期 + 运行时边界检查 | 编译期静态 + 运行时断言 | ZhC 提供跳过选项 |
| **整数溢出** | Debug panic / Release wrap | 可配置模式（wrap/saturate/panic） | ZhC 更灵活 |
| **泛型** | 泛型 + trait bound | `泛型<T>` + `约束` | 类似，ZhC 约束语法更宽松 |
| **线程安全** | `Send/Sync` trait | `共享型`/`线程独享型` 标注 | ZhC 用标注，Rust 用 trait |
| **模块系统** | `mod` + `use` + 可见性 | `模块` + `导入` + 可见性 | 类似，ZhC 针对中文优化 |
| **智能指针** | `Box`/`Rc`/`Arc`/`Weak` | `独享指针`/`共享指针`/`弱指针` | 语义一致，命名更直观 |
| **学习曲线** | 陡峭（生命周期是最大门槛） | **平缓**（中文关键字 + 渐进式检查） | ZhC 的核心优势 |
| **C 互操作** | FFI（unsafe） | 直接编译为 C ABI | ZhC 与 C 天然互操作 |
| **中文支持** | 无 | 完整中文关键字 | ZhC 独有 |
