## 核心关键词

### 1. 模块定义相关

| 中文关键词 | 英文对应 | 用途说明 | 示例 |
|:---|:---|:---|:---|
| **模块** | `namespace` | 定义一个新的模块/命名空间 | `模块 数学工具 { ... }` |
| **导入** | `import` 或 `using` | 导入其他模块的公开符号 | `导入 数学工具;` |
| **公开** | `public:` | 标记模块内公开访问的符号区域 | `公开: 函数 平方() { ... }` |
| **私有** | `private:` | 标记模块内私有访问的符号区域 | `私有: 静态 整数型 内部变量;` |

### 2. 访问修饰符

| 修饰符 | 可见性范围 | 转换规则 | 设计理由 |
|:---|:---|:---|:---|
| **公开:** | 模块外可见 | 转换为C++的`public:`或普通函数 | 提供API接口 |
| **私有:** | 仅模块内可见 | 转换为`static`函数/变量 | 隐藏实现细节 |
| **保护:** | 嵌套模块可见（可选） | 特殊前缀处理 | 支持模块内部封装 |

### 3. 符号引用语法

| 语法 | 说明 | 转换目标 |
|:---|:---|:---|
| `模块名::符号名` | 限定符号引用 | `模块名_符号名` |
| `导入 模块名;` | 导入模块 | `using namespace 模块名;` 或相应处理 |
| `别名 新名 = 模块名::原名;` | 创建别名（可选） | 宏定义或类型别名 |

## 详细语法规范

### 模块定义语法
```
模块 模块名 {
    [公开: | 私有: | 保护:]* 
    定义列表
}
```

**语法说明：**
1. `模块` 为关键字，后跟模块名标识符
2. 模块名遵循C标识符规则，支持中文字符
3. 模块体用 `{ }` 包裹
4. 模块内可包含多个访问控制区域
5. 每个区域以 `公开:`、`私有:`、`保护:` 开头

### 访问控制区域语法
```
公开:
    类型定义 | 函数定义 | 变量定义 | 常量定义

私有:
    类型定义 | 函数定义 | 变量定义 | 常量定义

保护:
    类型定义 | 函数定义 | 变量定义 | 常量定义
```

### 导入语句语法
```
导入 模块名;                    # 导入整个模块
导入 模块名::符号名;            # 导入特定符号
导入 模块名 为 别名;            # 导入并重命名（可选）
```

## 设计决策说明

### 1. 为什么使用`模块`而不是`命名空间`？
- **中文友好**：`模块`更符合中文开发者直觉
- **概念清晰**：`模块`暗示了封装和独立性
- **扩展性**：未来可扩展到真正的模块系统（不仅仅是命名空间）

### 2. 为什么使用`公开:`/`私有:`而不是关键字前缀？
- **可读性**：区域化声明更清晰易读
- **简洁性**：减少每个符号前的重复修饰符
- **一致性**：与面向对象语言的访问控制风格一致

### 3. 为什么支持嵌套模块？
```
模块 外层 {
    模块 内层 {
        公开: 函数 内部函数() { ... }
    }
}
```
- **组织性**：支持复杂项目的层级组织
- **封装性**：内层模块可对外层隐藏细节
- **兼容性**：为未来真正的模块系统做准备

## 语法示例

### 示例1：基础模块定义
```c
模块 数学计算 {
    公开:
        常量 PI = 3.1415926;
        
        函数 计算面积(参数 浮点型 半径) -> 浮点型 {
            返回 PI * 半径 * 半径;
        }
    
    私有:
        静态 整数型 调用次数 = 0;
        
        函数 记录调用() -> 空型 {
            调用次数++;
        }
}
```

### 示例2：模块导入与使用
```c
模块 主程序 {
    导入 数学计算;
    
    函数 主函数() -> 整数型 {
        浮点型 面积 = 数学计算::计算面积(5.0);
        打印("半径为5的圆面积: %f", 面积);
        返回 0;
    }
}
```

### 示例3：嵌套模块
```c
模块 图形库 {
    模块 二维 {
        公开:
            类 点 { 属性: x, y; }
            函数 绘制点(参数 点 p) -> 空型;
    }
    
    模块 三维 {
        公开:
            类 点 { 属性: x, y, z; }
            函数 绘制点(参数 点 p) -> 空型;
    }
}
```

## 转换规则设计

### 基础转换策略
1. **模块名映射**：为每个模块生成唯一前缀
2. **符号重命名**：公开符号添加模块名前缀
3. **访问控制转换**：
   - `公开:` → 普通函数/变量（带前缀）
   - `私有:` → `static`函数/变量（不带前缀）
4. **导入处理**：生成相应的`using`声明或宏定义

### 命名约定
- 模块前缀：`模块名_`（使用下划线分隔）
- 嵌套模块：`外层_内层_`
- 避免冲突：添加唯一标识符（可选）

## 兼容性考虑

### 与现有语法的兼容性
1. **不破坏现有代码**：模块语法是可选的扩展
2. **逐步迁移**：现有代码可继续工作
3. **工具链支持**：现有工具（IDE、调试器等）应能处理转换后的代码

### 与C/C++的兼容性
1. **可编译性**：转换后的代码应能被标准C编译器接受
2. **可调试性**：保留足够的调试信息（源码映射）
3. **性能影响**：转换不应引入显著性能开销
