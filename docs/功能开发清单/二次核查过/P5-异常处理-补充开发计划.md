## 三个关键风险及解决方案

### 风险一：IR 层断裂

**问题描述**：语义分析层 (semantic_analyzer.py) 的 _analyze_try_stmt 等方法只做类型检查，**不生成 IR**。IR 生成器 (ir_generator.py) 没有对应的 visitor 方法，导致 AST 节点无法转换为 IR 指令。

**现状**：
```python
# semantic_analyzer.py - 只做类型检查，不生成 IR
def _analyze_try_stmt(self, node: TryStmtNode):
    self._ensure_type(node.exception_type, 'Exception', ...)
    for clause in node.catch_clauses:
        self._analyze_catch_clause(clause)
    if node.finally_clause:
        self._analyze_finally_clause(node.finally_clause)
    # ❌ 没有调用 IR 生成器
```

**解决方案**：

在语义分析器与 IR 生成器之间建立协作机制：

1. 在 SemanticAnalyzer 中保存对 IRGenerator 的引用（self.ir_generator）
2. 在 _analyze_try_stmt 等方法末尾，当类型检查通过后，调用 self.ir_generator.visit_try_stmt(node) 生成 IR
3. 在 IRGenerator 中实现 visit_try_stmt、visit_throw_stmt 等方法，将 AST 节点转换为 IRTryBlock / IRThrow 等 IR 节点

**实施位置**：贯穿阶段一和阶段四。

---

### 风险二：策略缺失

**问题描述**：InstructionStrategyFactory.DEFAULT_STRATEGIES 只注册了普通指令策略，异常处理相关的策略（TryStrategy、CatchStrategy、ThrowStrategy、LandingPadStrategy、ResumeStrategy、InvokeStrategy）全部缺失，导致 emit_instruction 时遇到这些 opcode 会抛出 NotImplementedError。

**现状**：
```python
# llvm_instruction_strategy.py
class InstructionStrategyFactory:
    DEFAULT_STRATEGIES = {
        # ... 普通策略 ...
        # ❌ 没有 TRY, CATCH, THROW, RESUME, LANDINGPAD, INVOKE
    }
```

**解决方案**：

1. **前置调研**：在阶段二开始前，先验证 llvmlite 的 LLVM 版本是否支持 landingpad、resume、invoke 指令（见风险三）
2. **实现六策略类**：每个策略对应一个 Opcode，负责生成目标 IR
3. **注册到工厂**：DEFAULT_STRATEGIES 中补充六项，或在 LLVMLikeBackend.__init__ 中动态注册
4. **测试覆盖**：每个策略需要有单元测试验证生成的 IR 正确性

**新增任务**：见阶段二。

---

### 风险三：llvmlite 限制

**问题描述**：llvmlite 封装了特定版本的 LLVM，不同版本的 llvmlite 对异常处理指令的支持程度可能不同。需要提前验证以下指令是否可用：

- llvm.landingpad / landingpad 指令
- resume 指令
- invoke 指令
- llvm.eh.typeid.for intrinsics
- llvm.eh.begincatch / llvm.eh.endcatch intrinsics

**解决方案**：

在 **阶段二之前**，新增 **阶段 0（调研验证）**。

#### 阶段 0：llvmlite EH 能力调研与 POC

| 任务 | 优先级 | 说明 |
|------|--------|------|
| 调研 llvmlite 版本与 LLVM 版本对应关系 | P0 | 查看 llvmlite.llvm_version |
| 验证 landingpad 指令是否可调用 | P0 | 直接在 ir_builder 中测试 builder.landingpad(...) |
| 验证 resume 指令是否可用 | P0 | 检查 llvmlite.ir.LLVMContext 是否支持 |
| 验证 invoke 指令是否可调用 | P0 | builder.invoke(...) 的 API 存在性 |
| 验证 eh.typeid.for 是否为有效 intrinsics | P0 | 检查 llvmlite 是否暴露该函数 |
| 编写 POC 验证 IR 生成 | P0 | 生成包含 landingpad/resume 的最小 LLVM IR，验证语法正确 |
| 输出调研报告 | P0 | 汇总各指令支持情况，明确哪些需要 workaround |

**如果 llvmlite 不支持某些指令**：
- landingpad 不支持 → 考虑使用 C 运行时 setjmp/longjmp 模拟（已有 zhc_exception.h/c）
- invoke 不支持 → 检查是否有 builder.call + 间接分支的替代方案
- resume 不支持 → 可以省略（landingpad 的返回值直接用于 ret）
