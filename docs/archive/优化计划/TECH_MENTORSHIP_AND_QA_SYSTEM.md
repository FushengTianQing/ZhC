## 三、资深导师制 (Technical Mentorship)

### 3.1 导师职责定义

#### A. 代码审查导师 (Code Review Mentor)

**核心使命**: 每个合并到 main 的 PR 都必须经过导师审查。

| 审查维度 | 具体内容 | 优先级 |
|---------|---------|--------|
| 正确性 | 逻辑正确、边界处理、错误处理 | P0 必须通过 |
| 可读性 | 命名、结构、注释、方法长度 | P0 必须通过 |
| 性能 | 算法复杂度、热点路径、内存使用 | P1 应该通过 |
| 安全性 | 输入验证、注入防护、敏感数据 | P0 必须通过 |
| 可测试性 | 测试覆盖、测试质量、Mock 合理性 | P0 必须通过 |
| 架构 | 设计原则、耦合度、扩展性 | P1 应当关注 |

**审查响应 SLA**:
- 收到 PR 后 **4 小时内** 开始初审
- 初审意见 **24 小时内** 给出
- 修改后的复审 **12 小时内** 完成

#### B. 技术设计导师 (Design Mentor)

以下情况**必须**邀请导师参与设计讨论：

- [ ] 新增编译器 Pass（优化/分析阶段）
- [ ] 新增语言特性（语法/语义变更）
- [ ] 模块间接口的重大变更
- [ ] 数据结构的重新设计
- [ ] 性能关键路径的改动

**设计评审产出物**:
1. 设计文档（含动机、方案对比、影响范围）
2. 接口定义（类型签名、调用关系图）
3. 风险评估（兼容性、性能、测试策略）

#### C. 技术成长导师 (Growth Mentor)

**定期活动**:

| 活动 | 频率 | 形式 | 内容 |
|------|------|------|------|
| 代码走查 (Code Walkthrough) | 每周 1 次 | 1 小时 | 逐行阅读一个核心模块 |
| 技术分享会 | 双周 1 次 | 45 分钟 | 编译器技术/设计模式/最佳实践 |
| Bug 复盘 | 每次 P0/P1 Bug 修复后 | 30 分钟 | 根因分析、预防措施 |
| 1-on-1 | 每月 1 次 | 30 分钟 | 个人成长、困惑解答 |

## 四、代码质量门禁系统 (Quality Gate System)

### 4.1 分级质量标准

#### Level 0: 提交前自检（开发者自己跑）

```bash
#!/bin/bash
# pre-commit-check.sh - 每次提交前必须通过

set -e

echo "=== ZHC 质量门禁: Level 0 自检 ==="

# 1. 格式化检查
echo "[1/4] Black 格式化检查..."
black --check src/ tests/

# 2. Lint 检查
echo "[2/4] Ruff Lint 检查..."
ruff check src/ tests/

# 3. 类型检查
echo "[3/4] MyPy 类型检查..."
mypy src/ --ignore-missing-imports

# 4. 单元测试（仅改动相关模块）
echo "[4/4] 单元测试..."
python -m pytest tests/ -x -q

echo "✅ Level 0 自检全部通过！"
```

#### Level 1: PR 自动化检查（CI 执行）

```yaml
# .github/workflows/quality-gate.yml
name: Quality Gate

on:
  pull_request:
    branches: [main]

jobs:
  quality-gate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.9'

      - name: Install dependencies
        run: |
          pip install -e ".[dev]"
          pip install black ruff mypy pytest pytest-cov

      # === P0 门禁 ===

      - name: Black Format Check
        run: black --check src/ tests/

      - name: Ruff Lint
        run: ruff check src/ tests/

      - name: MyPy Type Check
        run: mypy src/ --ignore-missing-imports

      - name: Unit Tests
        run: pytest tests/ -x -q --tb=short

      # === P1 质量指标 ===

      - name: Coverage Report
        run: pytest tests/ --cov=src --cov-report=term-missing --cov-fail-under=15
        # 注意: 当前覆盖率仅 15%，逐步提升门槛

      - name: Complexity Check
        run: |
          # 使用 radon 检查圈复杂度
          pip install radon
          radon cc src/ -a -s | head -30
          # 目标: 平均圈复杂度 < 10, 单函数最高 < 20

      - name: Security Scan
        run: |
          pip install bandit
          bandit -r src/ -ll --skip-B101
```

#### Level 2: 人工审查卡点（导师执行）

PR 必须**全部满足**以下条件才能合并：

##### P0 - 一票否决项（任一不满足即拒绝）

| # | 检查项 | 验证方式 | 说明 |
|---|--------|---------|------|
| 1 | 所有自动化检查通过 | CI 绿色 | 黑屏不可合并 |
| 2 | 新代码有对应测试 | 查看 diff | 无测试不合并 |
| 3 | 覆盖率不下降 | CI 覆盖率报告 | 下降则需补充 |
| 4 | 无 P0 安全漏洞 | Bandit 扫描 + 人工 | 含漏洞必须修 |
| 5 | 异常处理完整 | 代码审查 | 不允许裸 except/pass |
| 6 | 文档字符串齐全 | 代码审查 | 公共 API 必须有 docstring |
| 7 | Commit 信息规范 | Git log | 符合 conventional commits |

##### P1 - 强烈建议项（需说明原因才可跳过）

| # | 检查项 | 标准 | 说明 |
|---|--------|------|------|
| 1 | 方法长度 | ≤ 40 行 | 超出需拆分或注释理由 |
| 2 | 圈复杂度 | ≤ 10 | 超出需说明且补充测试 |
| 3 | 类型注解 | 覆盖率 ≥ 80% | 公共 API 必须有注解 |
| 4 | 命名清晰度 | 见名知意 | 不允许缩写/拼音命名 |
| 5 | 无硬编码常量 | 使用具名常量 | 魔法数字提取为常量 |
| 6 | 无重复代码 | 复用 > 复制 | DRY 原则 |

##### P2 - 建议改进项（合并后记入 Tech Debt）

| # | 检查项 | 说明 |
|---|--------|------|
| 1 | 性能基准 | 热路径应有基准测试 |
| 2 | 架构图更新 | 结构变更应更新 ARCHITECTURE.md |
| 3 | 示例代码 | 用户可见功能应有示例 |
| 4 | 错误消息国际化 | 中文错误消息一致性 |

### 4.2 质量打分系统

每个 PR 合并时，导师进行质量打分（1-5 分）：

```python
class QualityScore:
    """PR 质量评分卡"""

    def __init__(self):
        self.categories = {
            "correctness": {  # 正确性, 权重 30%
                "logic_correct": 5,
                "edge_cases": 5,
                "error_handling": 5,
            },
            "readability": {  # 可读性, 权重 25%
                "naming": 5,
                "structure": 5,
                "comments": 5,
            },
            "testability": {  # 可测试性, 权重 25%
                "coverage": 5,
                "test_quality": 5,
                "boundary_tests": 5,
            },
            "architecture": {  # 架构, 权重 20%
                "design_principles": 5,
                "coupling": 5,
                "extensibility": 5,
            }
        }

    def calculate(self) -> dict:
        """计算加权总分"""
        # 总分 5 分制
        # ≥ 4.0: 优秀，直接合并
        # ≥ 3.0: 良好，小问题可后续处理
        # ≥ 2.0: 及格，需修改后重审
        # < 2.0: 不及格，需重大修改
        pass
```

# Parser 质量检查要点

class ParserQualityCheck:
    """Parser 代码质量检查清单"""

    - [ ] 递归下降的每个函数都有明确的语法规则引用

    # ✅ 可维护性

    - [ ] 语法规则变更只需改一处
    - [ ] Token 类型集中定义，不用魔法数字

    # ✅ 测试要求
    - [ ] 每个语法产生式至少 2 个正例
    - [ ] 每个 error recovery 至少 1 个测试
    - [ ] 错误消息包含位置信息（行:列）

### Phase 2: 覆盖率恢复战（第 2-3 周）

**目标**: 把覆盖率从 15% 拉回 50%+

这是当前最紧急的任务。新增的大量模块（SSA、数据流、循环优化、内联、泛型、模式匹配、异步、LSP、DWARF 等）都没有足够的测试覆盖。

**策略**: 按 **风险等级** 排序补测试

| 优先级 | 模块 | 当前预估覆盖 | 理由 |
|--------|------|-------------|------|
| P0 | src/semantic/ (泛型/模式匹配/异步) | ~30% | 核心语言特性，出错影响大 |
| P0 | src/ir/ (ssa/dataflow/loop/inline) | ~35% | 编译器正确性基础 |
