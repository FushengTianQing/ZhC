## 导入机制设计

### 1. 导入语法形式

#### 基础导入
```c
导入 模块名;                    // 导入整个模块的公开符号
```

#### 选择性导入
```c
导入 模块名::符号名;            // 导入特定符号
导入 模块名::{符号1, 符号2};     // 导入多个符号（可选）
```

#### 重命名导入
```c
导入 模块名 为 别名;            // 为模块创建别名
导入 模块名::符号名 为 新名;     // 为符号创建别名
```

### 2. 导入语义规则

#### 导入时机
- **编译时导入**：在预处理阶段解析和处理
- **静态导入**：不支持运行时动态导入
- **单次导入**：重复导入同一模块无副作用

#### 导入范围
- **文件作用域**：导入在当前文件有效
- **模块作用域**：导入在模块内有效（可选特性）
- **全局导入**：不支持全局导入，避免污染命名空间

### 3. 导入路径解析

#### 模块定位规则
1. **相对路径**：`导入 ./子目录/模块名;`
2. **绝对路径**：`导入 /项目根目录/模块/模块名;`
3. **标准库**：`导入 标准库::模块名;`

#### 搜索路径
```
1. 当前文件所在目录
2. 项目根目录
3. 标准库目录
4. 用户配置的额外路径
```

## 导出机制设计

### 1. 显式导出语法

#### 区域导出（主要方式）
```c
模块 数学工具 {
    公开:                     // 此区域内所有符号自动导出
        函数 平方() { ... }
        常量 PI = 3.14;
}
```

#### 显式导出声明（可选）
```c
模块 数学工具 {
    函数 内部函数() { ... }
    常量 内部常量 = 100;
    
    导出 内部函数 为 公开函数;   // 显式导出
    导出 内部常量;              // 使用原名导出
}
```

### 2. 导出控制

#### 可导出类型
- 函数、变量、常量、类型定义（结构体、枚举等）
- 嵌套模块（作为整体导出）

#### 不可导出类型
- 宏定义（由预处理器处理，不进入模块系统）
- 私有符号（`私有:`区域的符号）
- 局部符号（函数内部定义）

### 3. 模块接口文件（可选高级特性）

#### 接口声明文件（.zhi）
```c
// math.zhi - 接口文件
模块 数学工具 {
    公开:
        函数 平方(参数 整数型 x) -> 整数型;
        常量 PI;
}
```

#### 实现文件（.zhc）
```c
// math.zhc - 实现文件
模块 数学工具 {
    公开:
        函数 平方(参数 整数型 x) -> 整数型 {
            返回 x * x;
        }
        
        常量 PI = 3.1415926;
}
```

## 依赖管理

### 1. 依赖解析规则

#### 直接依赖
```c
// 文件A.zhc
模块 A {
    导入 B;    // A直接依赖B
}
```

#### 传递依赖
```c
// 文件A.zhc
模块 A {
    导入 B;    // A依赖B
}

// 文件B.zhc  
模块 B {
    导入 C;    // B依赖C
}
// 结果：A间接依赖C，但A不能直接使用C的符号
```

### 2. 循环依赖检测与处理

#### 检测规则
- 编译时检测模块间的循环依赖
- 记录依赖图，检测环

#### 处理策略
1. **禁止直接循环**：模块A导入B，同时B导入A → 编译错误
2. **允许间接循环**：通过第三个模块形成的间接依赖允许
3. **前向声明**：提供前向声明机制解决某些循环依赖

### 3. 依赖版本控制（未来扩展）

#### 基本版本语法
```c
导入 模块名 版本 "1.0.0";
导入 模块名 版本 ">=1.0.0 <2.0.0";
```

#### 版本冲突解决
- 同一模块不同版本不能同时导入
- 依赖解析失败时提示版本冲突

## 编译与链接

### 1. 模块编译流程

#### 单模块编译
```
数学工具.zhc → 预处理器 → 数学工具.c → 编译器 → 数学工具.o
```

#### 多模块编译
```
模块A.zhc → A.c → A.o
模块B.zhc → B.c → B.o
主程序.zhc → 主程序.c → 主程序.o
链接器 → 可执行文件（包含A.o + B.o + 主程序.o）
```

### 2. 符号链接处理

#### 公开符号链接
- 模块公开符号生成全局可见的C符号
- 链接器可正确解析跨模块引用

#### 私有符号处理
- 私有符号生成static符号，不参与链接
- 每个模块的私有符号独立，无冲突

### 3. 编译优化

#### 编译缓存
- 缓存已编译模块，避免重复编译
- 依赖变化时自动重新编译

#### 增量编译
- 只编译修改的模块及其依赖
- 减少大型项目的编译时间

## 错误处理机制

### 1. 导入错误

#### 模块未找到
```
错误：找不到模块 '不存在的模块'
建议：检查模块名拼写，或添加模块搜索路径
```

#### 符号未找到
```
错误：模块 '数学工具' 中没有符号 '不存在的函数'
建议：检查符号名，或查看模块的公开接口
```

### 2. 导出错误

#### 重复导出
```
错误：符号 '平方' 在模块 '数学工具' 中重复导出
建议：检查是否有重复定义
```

#### 私有符号导出
```
警告：尝试导出私有符号 '内部变量'
建议：将其移到公开区域，或移除导出声明
```

### 3. 依赖错误

#### 循环依赖
```
错误：检测到循环依赖：模块A → 模块B → 模块A
建议：重构设计，打破循环依赖
```

#### 版本冲突
```
错误：模块 '工具' 版本冲突：需要 1.0.0，但找到 2.0.0
建议：更新导入声明，或使用兼容版本
```

## 实现架构设计

### 1. 导入解析器设计

#### 解析流程
```
源代码 → 词法分析 → 识别导入语句 → 解析导入参数 → 
验证模块存在 → 加载模块接口 → 更新符号表
```

#### 数据结构
```python
class ImportStatement:
    def __init__(self):
        self.module_name = ""      # 模块名
        self.symbols = []          # 导入的符号列表（空表示全部）
        self.alias = None          # 别名
        self.location = None       # 导入位置
        
class ModuleInterface:
    def __init__(self):
        self.name = ""             # 模块名
        self.exports = {}          # 导出符号表
        self.dependencies = []     # 依赖模块列表
        self.file_path = ""        # 模块文件路径
```

class DependencyGraph:

        """添加依赖关系"""

### 3. 符号表集成

#### 符号解析流程
```
引用符号 → 查找当前作用域 → 未找到则查找导入符号 →

```

## 测试用例设计

### 用例1：基础导入导出
```c
// math.zhc
模块 数学 {
    公开: 函数 平方(x) { 返回 x*x; }
}

// main.zhc
模块 主程序 {
    导入 数学;
    
    函数 主函数() {
        打印("%d", 数学::平方(5));  // 输出: 25
    }
}
```

### 用例2：选择性导入
```c
模块 工具 {
    公开: 
        函数 工具1() { ... }
        函数 工具2() { ... }
        函数 工具3() { ... }
}

模块 用户 {
    导入 工具::{工具1, 工具3};
    
    函数 使用() {
        工具1();  // ✓ 可访问
        工具2();  // ✗ 未导入
        工具3();  // ✓ 可访问
    }
}
```

### 用例3：别名导入
```c
模块 很长名字的模块 {
    公开: 函数 复杂函数() { ... }
}

模块 用户 {
    导入 很长名字的模块 为 短名;
    
    函数 使用() {
        短名::复杂函数();  // 使用别名
    }
}
```

### 用例4：循环依赖检测
```c
// 错误示例
模块 A {
    导入 B;
    公开: 函数 a() { B::b(); }
}

模块 B {
    导入 A;  // 错误：循环依赖
    公开: 函数 b() { A::a(); }
}
```

## 设计决策记录

### 决策1：导入语法风格
- **选项A**：`使用 模块名`（类似C#）
- **选项B**：`导入 模块名`（类似Python/Java）
- **选项C**：`包含 模块名`（类似C++）
- **选择**：选项B（`导入 模块名`）
- **理由**：符合中文直觉，与现有`包含`（#include）区分

### 决策2：导出机制
- **选项A**：显式导出声明（每个符号单独导出）
- **选项B**：区域导出（通过`公开:`区域）
- **选项C**：混合模式（区域导出为主，显式导出为辅）
- **选择**：选项C（混合模式）
- **理由**：既方便（区域导出）又灵活（显式导出）

### 决策3：循环依赖处理
- **选项A**：完全禁止
- **选项B**：允许但警告
- **选项C**：提供前向声明机制
- **选择**：选项A（完全禁止）
- **理由**：简化实现，鼓励良好设计

## 兼容性考虑

### 与#include的兼容性
- `#include` 继续工作，处理头文件包含
- `导入` 用于模块系统，更高级的依赖管理
- 两者可共存，但建议新代码使用模块系统

### 向后兼容性
- 现有代码无需修改
- 模块系统是可选的扩展
- 可逐步迁移到模块系统
