## 寄存器分配器架构重构完成

### 背景
用户提出架构问题：寄存器分配器放在 `codegen` 模块，后续废弃 `codegen` 时会发生什么？

### 问题分析
1. **模块定位错误**：寄存器分配是 IR 层优化，不应该属于代码生成层
2. **依赖关系混乱**：`codegen` 模块依赖 `zhc.parser`，导致循环导入风险
3. **迁移成本高**：废弃 `codegen` 时需要修改所有导入路径

### 重构方案
将寄存器分配器从 `codegen` 移动到正确的架构位置：

```
src/
├── ir/                       # IR 层优化
│   ├── ssa.py               # SSA 构建
│   ├── dataflow.py          # 数据流分析
│   ├── loop_optimizer.py    # 循环优化
│   ├── inline_optimizer.py  # 内联优化
│   └── register_allocator.py # 寄存器分配 ← 移到这里
│
├── backend/                  # 后端层
│   ├── __init__.py          # 后端模块初始化
│   └── allocator_interface.py # 后端适配接口
│
└── codegen/                  # 临时模块（已废弃）
    ├── __init__.py          # 废弃警告 + 重新导出
    ├── register_allocator.py # 废弃包装器
    └── allocator_interface.py # 废弃包装器
```

### 实施步骤
1. ✅ 创建 `src/backend/` 目录结构
2. ✅ 移动 `register_allocator.py` 到 `src/ir/`
3. ✅ 移动 `allocator_interface.py` 到 `src/backend/`
4. ✅ 更新所有导入路径
5. ✅ 更新测试文件
6. ✅ 添加废弃警告到 `codegen` 模块
7. ✅ 运行测试验证重构
8. ✅ 更新文档

## LLVM 后端集成方案讨论

### 背景
用户询问 ZhC 接入 LLVM 的具体方案，并讨论了以下问题：
1. C Backend 的两个文件关系（`c_backend.py` vs `c_codegen.py`）
2. ZhC IR 连接 C Backend 的意义
3. C 本身是前端，不是后端
4. Clang 的后端架构
5. GCC vs LLVM 发展对比

### 关键发现
1. **C Backend 不是真正的后端**
   - `src/ir/c_backend.py`: IR → C 代码生成器（dispatch table 模式）
   - `src/codegen/c_codegen.py`: AST → C 代码生成器（ASTVisitor 模式）
   - 两者都需要 gcc/clang 才能生成机器码
   - C Backend 实际上是"代码生成器"，不是"编译器后端"

2. **LLVM 后端架构**
   - 前端（Clang）→ IR（LLVM IR）→ 中间优化 → 后端（目标代码）
   - 后端流程：指令选择（SelectionDAG）→ 寄存器分配 → 代码发射
   - llvmlite: Python 绑定，支持 IR 生成、bitcode、JIT 执行

3. **GCC vs LLVM**
   - LLVM: 模块化、现代架构、活跃社区、工具链丰富
   - GCC: 成熟稳定、广泛支持、性能优化强、历史包袱
   - LLVM 更适合新项目集成

### LLVM 集成方案
**技术选型**: `llvmlite>=0.39.0`（LLVM 12 绑定）

**实施计划**（5 阶段，约 9.5 天）：

   - 创建 `src/backend/llvm_backend.py`
   - 使用 llvmlite API 生成真正的 LLVM IR
   - 支持 bitcode 输出和 JIT 执行

3. **阶段 3**: 类型映射（1 天）
   - ZhC 类型 → LLVM 类型映射表
   - 整数、浮点、数组、结构体、指针

   - ZhC IR Opcode → LLVM IR 指令映射
   - 算术、逻辑、控制流、函数调用
